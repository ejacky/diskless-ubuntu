# 无盘多系统平台 — 扩展性研究

> 本文档记录从 1~5 台（当前场景）扩展到 10~50 台、100 台，以及从千兆升级到万兆网络的远期技术研究和规划。
>
> 当前实现请以 [`ARCHITECTURE.md`](ARCHITECTURE.md) 为准。

---

## 1. 研究目标

| 阶段 | 规模 | 网络 | 存储 | 目标 |
|------|------|------|------|------|
| 当前 | 1~5 台 | 千兆 | 单服务端单盘 | MVP 验证，见 `ARCHITECTURE.md` |
| 扩展一 | 10~30 台 | 千兆→万兆 | 单服务端升级 | 支撑一个小型机房 |
| 扩展二 | 50~100 台 | 万兆/25GbE | 多存储服务器 | 支撑网吧或大型机房 |

## 2. 10~50 台扩展方案

### 2.1 网络升级路径

**千兆网络的极限**：
- 理论带宽：1 Gbps ≈ 125 MB/s
- 实际可用：约 80~100 MB/s（TCP/IP 开销、交换机延迟）
- 5 台并发启动 ≈ 400~500 MB/s = 3.2~4 Gbps → **已超千兆**

**升级选项**：

| 方案 | 带宽 | 成本 | 适用场景 |
|------|------|------|----------|
| **双千兆 LACP** | 2 Gbps | 低（现有网卡+交换机支持即可） | 10~15 台过渡方案 |
| **单万兆（10GbE）** | 10 Gbps | 中（网卡+交换机） | 15~30 台 |
| **双万兆 LACP** | 20 Gbps | 中高 | 30~50 台 |
| **25GbE/40GbE** | 25/40 Gbps | 高 | 50~100 台 |

**推荐演进**：千兆 → 双千兆聚合（立即可做，零额外硬件） → 单万兆（10~30台时） → 双万兆（30~50台时）。

### 2.2 单服务器性能模型（支撑 30 台）

假设 30 台同时开机（最坏情况），每台读取 50 MB/s：

| 资源 | 需求 | 推荐配置 |
|------|------|----------|
| **网络** | 30 × 50 = 1,500 MB/s ≈ 12 Gbps | 双万兆 LACP = 20 Gbps |
| **存储读 I/O** | 1,500 MB/s | NVMe SSD RAID-0（读速 3GB/s+） |
| **内存缓存** | 母盘热数据缓存 | 64GB+ RAM |
| **CPU** | iSCSI + LVM + TCP/IP | 8 核+ |

**实际场景**：30 台不会同时开机， staggered boot（错峰启动）可将并发降到 5~10 台。

### 2.3 读热点缓存优化

30 台同时启动 Windows/Linux 时，读取的**大多是母盘相同的数据块**（启动文件、系统 DLL、内核等）。

**利用 Linux 页缓存**：
```bash
# 在服务端，预先将母盘数据读入内存缓存
vmtouch -t /dev/diskless-vg/win10-golden
vmtouch -t /dev/diskless-vg/ubuntu-golden

# 验证缓存状态
vmtouch -v /dev/diskless-vg/win10-golden
```

**或使用 bcache / dm-cache**：如果 SSD 容量不足以放所有母盘，可以用 SSD 作为 HDD 的缓存层。

## 3. 100 台扩展方案

### 3.1 多存储服务器架构

单台服务器无论多强，都有 I/O 天花板。100 台需要**多台存储服务器**分担负载。

```
                            ┌─────────────────────────────┐
                            │      核心交换机（万兆）       │
                            │  （支持 LACP 链路聚合）        │
                            └─────────────┬───────────────┘
                                          │
                    ┌─────────────────────┼─────────────────────┐
                    │                     │                     │
            ┌───────▼───────┐     ┌───────▼───────┐     ┌───────▼───────┐
            │  接入交换机   │     │  接入交换机   │     │  接入交换机   │
            │  （千兆下行） │     │  （千兆下行） │     │  （千兆下行） │
            │  （万兆上联） │     │  （万兆上联） │     │  （万兆上联） │
            └───────┬───────┘     └───────┬───────┘     └───────┬───────┘
                    │                     │                     │
              30台客户端            30台客户端            40台客户端

┌─────────────────────────────────────────────────────────────────────────────┐
│                              存储服务器集群                                   │
│                                                                              │
│  ┌─────────────────┐  ┌─────────────────┐  ┌─────────────────┐             │
│  │   存储节点 A     │  │   存储节点 B     │  │   存储节点 C     │             │
│  │  (服务 30 台)    │  │  (服务 30 台)    │  │  (服务 40 台)    │             │
│  │                  │  │                  │  │                  │             │
│  │  • 双口 25GbE    │  │  • 双口 25GbE    │  │  • 双口 25GbE    │             │
│  │  • NVMe SSD      │  │  • NVMe SSD      │  │  • NVMe SSD      │             │
│  │  • 128GB RAM     │  │  • 128GB RAM     │  │  • 128GB RAM     │             │
│  │  • 独立 LVM Pool │  │  • 独立 LVM Pool │  │  • 独立 LVM Pool │             │
│  │  • 独立 Boot API │  │  • 独立 Boot API │  │  • 独立 Boot API │             │
│  └─────────────────┘  └─────────────────┘  └─────────────────┘             │
│                                                                              │
│  镜像同步：golden images 通过 rsync / zfs send 在节点间同步                    │
│  管理集中：统一的管理界面控制所有节点（阶段 3）                                  │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 3.2 单存储节点性能模型（支撑 30~40 台）

| 资源 | 需求 | 推荐配置 |
|------|------|----------|
| **存储读 I/O** | 40 × 50 = 2,000 MB/s | NVMe RAID-0/10，读速 4GB/s+ |
| **网络带宽** | 2,000 MB/s ≈ 16 Gbps | 双口 25GbE 聚合 = 50 Gbps |
| **内存缓存** | 缓存完整母盘 | 128GB~256GB RAM |
| **CPU** | iSCSI + LVM + TCP/IP | 16 核+（AMD EPYC / Intel Xeon） |

### 3.3 客户端分组策略

100 台不再由单一 Boot API 管理，而是按分组连接到不同存储节点：

```python
# 分组配置示例
clusters = {
    "cluster-a": {
        "server_ip": "192.168.31.10",
        "clients": ["aa:bb:cc:*", "11:22:33:*"],  # MAC 前缀匹配
        "capacity": 30,
    },
    "cluster-b": {
        "server_ip": "192.168.31.11",
        "clients": ["44:55:66:*", "77:88:99:*"],
        "capacity": 30,
    },
    # ...
}
```

## 4. 万兆网络规划

### 4.1 万兆 vs 千兆容量对比

| 指标 | 千兆（1GbE） | 万兆（10GbE） | 双万兆（20GbE） |
|------|-------------|--------------|----------------|
| 理论带宽 | 125 MB/s | 1,250 MB/s | 2,500 MB/s |
| 实际可用 | ~100 MB/s | ~1,000 MB/s | ~2,000 MB/s |
| 支撑并发启动台数 | 2~3 台 | 20~25 台 | 40~50 台 |
| 单客户端峰值 | 100 MB/s | 1,000 MB/s | 1,000 MB/s |

### 4.2 硬件选型建议

| 组件 | 入门万兆 | 主流万兆 | 高端（25GbE） |
|------|----------|----------|--------------|
| **网卡** | Intel X520/X540（光口/电口） | Mellanox ConnectX-4 LX | Mellanox ConnectX-5 |
| **交换机** | TP-LINK TL-ST1008F | MikroTik CRS312-4C+8XG | Arista / Cisco |
| **线缆** | DAC 直连铜缆（3m 内） | 光纤 + 光模块 | 光纤 + 光模块 |
| **成本** | 低 | 中 | 高 |

> **注意**：电口万兆（10GBASE-T）兼容性最好但功耗高、延迟稍大；光口/SFP+ 延迟低但需要光模块。

### 4.3 网络拓扑分离（推荐）

大规模场景下，建议将**管理流量**和**存储流量**分离：

```
┌─────────────────┐
│   管理网络      │ 千兆 ──► DHCP、Boot API HTTP、SSH 管理
│  (VLAN 10)     │
└─────────────────┘
         │
┌─────────────────┐
│   存储网络      │ 万兆 ──► iSCSI 数据传输（独立交换机或 VLAN）
│  (VLAN 20)     │
└─────────────────┘
```

**好处**：
- 存储流量不挤占管理带宽
- iSCSI 可配置独立网段和 MTU（建议 9000 Jumbo Frames）
- 故障隔离，易排查

## 5. 从 1~5 台扩展到 10~50 台的迁移路径

### 5.1 平滑升级策略

```
阶段 0（当前）：1~5 台，千兆，单服务器，单盘 SATA SSD
    │
    ▼ 增加客户端到 6~10 台
阶段 1：6~10 台，双千兆 LACP，单服务器，SATA SSD → NVMe SSD
    │
    ▼ 增加客户端到 10~20 台
阶段 2：10~20 台，单万兆（10GbE），单服务器，NVMe SSD，大内存
    │
    ▼ 增加客户端到 20~30 台
阶段 3：20~30 台，双万兆 LACP，单服务器，多 NVMe RAID，128GB+ 内存
    │
    ▼ 增加客户端到 30~100 台
阶段 4：30~100 台，多存储服务器，核心+接入交换机，25GbE+
```

### 5.2 各阶段关键变更

| 阶段 | 网络变更 | 存储变更 | 软件变更 |
|------|----------|----------|----------|
| 0→1 | 双千兆聚合 | 更换为 NVMe SSD | 无 |
| 1→2 | 升级万兆网卡+交换机 | 单 NVMe → NVMe RAID | 启用 vmtouch 缓存 |
| 2→3 | 双万兆聚合 | 多 NVMe RAID-0 | Boot API 增加性能监控 |
| 3→4 | 核心+接入架构 | 多服务器 LVM Pool | 增加集群管理、镜像同步 |

### 5.3 数据迁移

从单服务器扩展到多服务器时，golden images 需要同步：

```bash
# 节点间同步 golden image（使用 zfs send 或 rsync）
# 方案 A：rsync（简单）
rsync -avP /dev/diskless-vg/ubuntu-golden root@storage-b:/dev/diskless-vg/ubuntu-golden

# 方案 B：ZFS（更优，支持增量发送）
# 如果后端存储改用 ZFS：
zfs send tank/golden/ubuntu@snapshot | ssh storage-b zfs receive tank/golden/ubuntu
```

## 6. 关键技术预研

### 6.1 备选存储后端对比

| 后端 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **LVM Thin**（当前） | 内核原生、简单、与 targetcli 无缝 | 无内置网络复制 | 1~50 台 |
| **ZFS** | 压缩、去重、快照、send/receive | 内存 hungry，学习曲线 | 50~100 台 |
| **Ceph RBD** | 分布式、高可用、自动复制 | 复杂，需要 3+ 节点 | 100 台+ |
| **NFS + overlayfs** | 简单，文件级 | 性能差，不适合 Windows | Linux  only |

**结论**：
- 1~50 台：LVM Thin 足够
- 50~100 台：评估 ZFS（单节点性能更好）或 Ceph（分布式高可用）

### 6.2 iSCSI vs AoE vs NBD

| 协议 | 复杂度 | 性能 | 适用场景 |
|------|--------|------|----------|
| **iSCSI**（当前） | 中 | 高 | 通用，Windows/Linux 都支持 |
| **AoE (ATA over Ethernet)** | 低 | 很高 | Linux only，需要特殊网卡支持 |
| **NBD (Network Block Device)** | 低 | 中 | Linux only |

**结论**：由于必须支持 Windows，iSCSI 是唯一可行选择。

### 6.3 客户端本地缓存（阶段 4 可选）

100 台场景下，即使万兆网络也可能成为瓶颈。此时可考虑：

| 方案 | 原理 | 代价 |
|------|------|------|
| **bcache** | 客户端本地 SSD 作为 iSCSI 盘的读缓存 | 客户端需要 SSD |
| **Intel CAS** | 类似的缓存加速方案 | 非开源 |
| **LVM cache** | 服务端 SSD 缓存热点数据 | 服务端需要更多 SSD |

**网吧行业实际做法**：客户端保留 128G~256G SSD，系统从网络"流式"加载到本地 SSD 缓存，类似网维大师。这**不是纯无盘**，但兼容性和性能最好。

## 7. 总结：何时启动扩展

| 信号 | 行动 |
|------|------|
| 5 台同时开机明显变慢（>3分钟进系统） | 评估双千兆聚合或错峰启动 |
| 计划扩展到 10 台以上 | 升级 NVMe SSD + 万兆网卡 |
| 计划扩展到 30 台以上 | 设计多存储服务器架构 |
| thin pool 空间使用率 >80% | 扩容物理存储或增加服务器 |
| 单台服务器 CPU/内存持续 >80% | 增加服务器节点 |

> **原则**：不要在 1~5 台阶段过度设计。先让 MVP 跑起来，根据实际瓶颈逐步升级。
